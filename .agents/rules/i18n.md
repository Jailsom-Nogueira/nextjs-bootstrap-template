# Internationalization (i18n)

## Default

This template ships WITHOUT i18n. Most projects are English-only and `next-intl` adds complexity.

## When to add it

- More than one target locale.
- User-facing strings need translator workflow.

## Recommended setup

- `next-intl` (App Router-native, type-safe messages, server-component friendly)
- Locale routing via `/[locale]/...` segment.
- Messages in `messages/<locale>.json`.

## Rules (once added)

- NEVER hardcode user-facing strings in components. Always go through `t("key")`.
- Date / number formatting: use `Intl.DateTimeFormat` / `Intl.NumberFormat`, locale-aware.
- Pluralization: `t("items", { count })` with ICU MessageFormat.
- Fallback locale: `en`.
