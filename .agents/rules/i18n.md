# Internationalization (i18n)

This template ships with **next-intl v4** wired for three locales: `en` (en-us, default), `pt` (pt-br), `es` (es-es).

## File locations

- `src/i18n/routing.ts` тАФ locale list + `localePrefix: "as-needed"`.
- `src/i18n/navigation.ts` тАФ `Link`, `redirect`, `usePathname`, `useRouter`, `getPathname` (locale-aware).
- `src/i18n/request.ts` тАФ `getRequestConfig`, loads `messages/<locale>.json` per request.
- `messages/en.json`, `messages/pt.json`, `messages/es.json` тАФ translation bundles.
- `src/app/[locale]/layout.tsx` тАФ wraps the tree in `NextIntlClientProvider`.
- `src/middleware.ts` тАФ composes `createMiddleware(routing)` with the Supabase session refresh.

## Reading translations

**Server components (preferred):**

```tsx
import { getTranslations } from "next-intl/server";
const t = await getTranslations("common");
return <h1>{t("appName")}</h1>;
```

**Client components:**

```tsx
"use client";
import { useTranslations } from "next-intl";
const t = useTranslations("common");
return <button>{t("save")}</button>;
```

## Namespacing convention

Top-level keys group related strings:

- `common.*` тАФ buttons, generic verbs, loading/error states
- `nav.*` тАФ navigation labels
- `auth.*` тАФ login, signup, password
- `theme.*` тАФ light/dark/system
- `errors.*` тАФ generic error messages
- `home.*`, `admin.*`, `<feature>.*` тАФ feature-scoped
- `locale.*` тАФ language switcher labels

If a feature grows beyond ~15 keys, give it its own namespace.

## ICU plurals

```json
{ "items": "{count, plural, =0 {No items} one {# item} other {# items}}" }
```

```tsx
t("items", { count });
```

## Locale-aware formatting

Use next-intl's `format` helpers (server) or `useFormatter` (client) тАФ they pick up the active locale automatically.

```tsx
import { useFormatter } from "next-intl";
const format = useFormatter();
format.dateTime(date, { dateStyle: "medium" });
format.number(amount, { style: "currency", currency: "USD" });
```

Avoid raw `Intl.DateTimeFormat` unless you need the locale to differ from the request locale.

## Navigation

ALWAYS import navigation primitives from `@/i18n/navigation`, never from `next/link` or `next/navigation`. This ensures the locale prefix is preserved on transitions.

```tsx
import { Link, redirect, useRouter } from "@/i18n/navigation";

<Link href="/dashboard">{t("nav.dashboard")}</Link>;
router.push("/login");
redirect("/admin");
```

## Adding a new locale

1. Add the code (e.g. `fr`) to `routing.ts` `locales`.
2. Create `messages/fr.json` with the same key shape as `en.json`.
3. Add a label in `locale.fr` for the switcher.
4. That's it тАФ the layout, middleware, and switcher pick it up automatically.

## Rules

- **NEVER** hardcode user-facing strings. Even `"Save"` goes through `t("common.save")`.
- **NEVER** import from `next/link` or `next/navigation` in app code тАФ use `@/i18n/navigation`.
- **ALWAYS** add the key to all three locales (`en`, `pt`, `es`) in the same commit. Translation review can come later, but the key must exist or the build will warn.
- **ALWAYS** call `setRequestLocale(locale)` in server components inside `[locale]` segments to enable static rendering.

## Fallbacks

- Default locale: `en`.
- Missing key behavior: next-intl throws in development, falls back to the key name in production. Run `npm run build` before merging to catch missing keys.
